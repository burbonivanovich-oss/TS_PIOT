# /monitor-competitors — мониторинг конкурентного контента

Дополнение к `/monitor-rss`. Где `monitor-rss` смотрит НПА и
разъяснения ведомств (первоисточники), `monitor-competitors` смотрит
**аналитические агрегаторы** (Клерк, Бухонлайн, E-Pepper и т.д.) —
не для копирования тем, а как индикатор «куда смещается интерес
аудитории МСБ».

**Главное правило: не дублировать темы дословно.** Конкурент-аналитика
— это **trend detector**, не источник идей для копирования. Если все
агрегаторы пишут про «новые штрафы за маркировку обуви» — это сигнал,
что у нас должна быть обновлённая или новая своя статья с
**отличающимся углом** (свой research-brief, свои примеры, свои
интервью). Дословное копирование заголовков запрещено.

Вызов:
- `/monitor-competitors` — окно 14 дней, все кластеры
- `/monitor-competitors --window 30d` — для среднесрочного планирования
- `/monitor-competitors --cluster markirovka` — только маркировка

## Алгоритм

### Шаг 1 — Загрузка источников

Прочитать `src/content/wiki/rss-sources.md`. Отобрать только
**агрегаторы и аналитику** (тип источника = «agregator» или
«analytics»):

- Клерк.ру (все ленты)
- Бухонлайн
- E-Pepper.ru
- Retailer.ru
- Frontoffice (HoReCa)
- Banki.ru / Bankir.ru
- Главбух
- Право.ру

Регуляторы (`nalog.gov.ru`, `честныйзнак.рф`, `pravo.gov.ru`) — **не**
смотрим в этом скилле. Их курирует `/monitor-rss`.

### Шаг 2 — Получение и нормализация

Для каждой ленты:
1. WebFetch RSS, окно по `--window` (по умолчанию 14d).
2. Извлечь заголовки, даты, краткие анонсы.
3. Нормализовать тему: убрать кликбейт, привести к нейтральному
   формулировку («Каких касс ждать?» → «новые модели касс»).

### Шаг 3 — Кластеризация по теме

Сгруппировать материалы по нашим контент-кластерам через ключевые
слова. Если фильтр `--cluster` — оставить только нужный.

Внутри каждого кластера — подгруппы по нормализованной теме.
Параметры кластера:
- **Кол-во материалов** в окне — индикатор «горячести темы».
- **Доминирующий угол** — что именно акцентируют конкуренты (штрафы /
  сроки / процедура / кейсы).
- **Источники** — какие площадки пишут.

### Шаг 4 — Сравнение с нашим контентом

Для каждой темы:
1. Прочитать frontmatter всех `src/content/blog/*.md`.
2. Найти статьи с пересечением по ключам (нормализованный title +
   seo.keywords).
3. **Применить пороги дублей из `/monitor-rss` Шаг 4:**
   - ≥ 0.9 → у нас уже есть статья → пометить «не делать»
   - 0.6–0.9 → у нас близкая статья → «возможный апдейт» с указанием
     slug
   - < 0.6 → у нас нет такого угла → «потенциальная новая статья»

### Шаг 5 — Trend signals

Дополнительно выделить:
- **Темы с резким ростом упоминаний** (≥ 5 материалов за окно у разных
  источников) — это конкурентный спрос, который может стать
  поисковым трендом раньше, чем Wordstat зафиксирует.
- **Кейсы**: разборы реальных ситуаций МСБ (споры с ФНС, штрафы,
  переходы). Это материал для нашего формата «разборы кейсов» (см.
  #33 в backlog).
- **Методологические дискуссии**: разные интерпретации одной нормы.
  Это сигнал что норма неоднозначна — у нас должно быть чёткое
  объяснение.

### Шаг 7 — Сохранение в trend-журнал

Если пользователь подтверждает («да», «сохрани»):
1. Дописать в `src/content/wiki/competitor-trends.md` блок с датой
   отчёта и таблицей горячих тем.
2. Закоммитить: `chore: trend-журнал DD.MM.YYYY`.

Это журнал для исторического анализа: если тема была «горячей» 3
месяца назад и мы её закрыли — отметить в журнале как «реализовано».
Помогает не возвращаться к одному и тому же.

## Чем отличается от `/monitor-rss`

| | `/monitor-rss` | `/monitor-competitors` |
|---|---|---|
| Источники | НПА, регуляторы, ведомства | Агрегаторы, аналитика |
| Цель | новые темы / обновления по нормам | trend detector, кейсы |
| Куда идёт | `/plan-content` или `/maintain-content` | `competitor-trends.md` |
| Что нельзя | пропустить новую норму | копировать темы дословно |

---

## Правила

- **Не копировать темы дословно.** Конкурент-аналитика — для понимания
  спроса, не для рерайта.
- **Не публиковать без research-brief'а** через `/create-article`.
- Если в `competitor-trends.md` тема числится «уже закрыта» — не
  выводить её снова, пока не пройдёт 90 дней.
- Не путать с НПА: если в анонсе агрегатора есть ссылка на конкретный
  закон/ПП, эту норму надо ставить **через `/monitor-rss`**, не через
  этот скилл.
- Язык отчёта — только русский.
