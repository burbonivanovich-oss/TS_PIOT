# /publish-article — гейтированный пайплайн статьи

Запускает все стадии создания статьи последовательно. Каждая стадия — шлюз:
провал → автоматические правки → повтор (максимум 2 раза на стадию).
После двух неудачных попыток — остановка с объяснением причины.

Вызов: `/publish-article "штраф за работу без ТС ПИоТ в 2026"`

---

## Перед запуском

Вывести статус:
```
▶ /publish-article "<тема>"
━━━━━━━━━━━━━━━━━━━━━━━━━━━
Стадия 1/8: DISCOVERY
```

---

## Стадия 1 — DISCOVERY

**Цель:** убедиться, что тема актуальна, не дублирует существующий контент,
и собрать факты для написания.

**Действия:**

1. Прочитать `src/content/wiki/content-plan-2026.md` — найти тему в плане,
   уточнить целевой запрос и приоритет.

2. Проверить дублирование по существующим статьям:
   - Прочитать frontmatter всех файлов в `src/content/blog/`
   - Сравнить тему с существующими title и seo.keywords
   - Если пересечение > 70% по смыслу → СТОП

3. Запустить агент `research-specialist` с темой.
   Сохранить research brief в `.claude/research/YYYY-MM-DD-slug-brief.md`.

**Шлюз 1:**
```
[ ] Тема отсутствует или существенно отличается от существующих статей?
[ ] Research brief содержит минимум 3 источника с НПА/датами?
[ ] Есть чёткий целевой поисковый запрос?
```

**При провале:**
- Если тема дублирует существующую → СТОП, вывести:
  ```
  ✗ СТОП: тема пересекается с [slug] на X%.
  Варианты:
    1. Переформулировать угол: "[предложение]"
    2. Дополнить существующую статью [slug]
  ```
- Если нет источников → уточнить формулировку и повторить research.

**Выход:** `.claude/research/YYYY-MM-DD-slug-brief.md`

---

## Стадия 2 — WRITE

**Цель:** написать черновик статьи по research brief.

**Действия:**

Запустить агент `content-writer` с research brief.

Требования к черновику:
- Длина: 1200–2500 слов
- Структура: лид → 5–7 H2 → FAQ (минимум 4 вопроса) → заключение
- Минимум 3 внутренних ссылки на `/blog/`, `/category/`, `/kalkulyator-shtrafov/`
- Минимум 1 ссылка на НПА (consultant.ru, честныйзнак.рф, nalog.gov.ru)
- Frontmatter заполнен полностью по шаблону из CLAUDE.md
- `draft: true`

Сохранить в `src/content/blog/YYYY-MM-DD-slug.md`.

**Шлюз 2:**
```
[ ] Файл создан и парсится без ошибок?
[ ] Frontmatter содержит: title, description, pubDate, tags, categories, draft, seo.keywords?
[ ] Объём тела статьи ≥ 1000 слов?
```

Проверка объёма:
```python
python3 -c "
content = open('src/content/blog/SLUG.md').read()
parts = content.split('---')
body = '---'.join(parts[2:])
print(len(body.split()), 'слов')
"
```

**При провале:** content-writer получает список конкретных замечаний → правки → повтор.

**Выход:** `src/content/blog/YYYY-MM-DD-slug.md` (draft: true)

## Стадия 3 — FACTCHECK

**Цель:** убедиться, что все цифры и нормы проверяемы.

**Действия:**

Запустить `/blog factcheck src/content/blog/SLUG.md`.

**Шлюз 3:**
```
[ ] Нет claims с confidence < 0.7 без пометки [нужна проверка]?
[ ] Все даты сроков подтверждены источником?
[ ] Все суммы штрафов соответствуют актуальной редакции КоАП/НПА?
```

**При провале:** вывести таблицу слабых claims:
```
Claim: "штраф до 300 000 ₽"    confidence: 0.5    источник: не указан
Claim: "срок с 01.03.2025"      confidence: 0.8    источник: Постановление № 1944
```
Агент правок получает конкретный список → исправляет или ставит [нужна проверка] → повтор.

Максимум 2 попытки. После второй провала — СТОП с пометкой:
```
⚠ СТОП: factcheck не пройден. Требуется ручная проверка:
  - [список проблемных утверждений]
Установите [нужна проверка] вручную и запустите снова.
```

**Выход:** статья с подтверждёнными фактами или явными пометками [нужна проверка]

## Стадия 4 — РЕДПОЛИТИКА

**Цель:** убедиться, что статья соответствует стандартам портала.

**Действия:**

Проверить статью по чеклисту вручную (читать frontmatter + структуру):

**Шлюз 4:**
```
Объём и структура:
[ ] Тело статьи ≥ 800 слов (сателлит) или ≥ 1500 (опорная)
[ ] Есть минимум 4 секции H2
[ ] Есть FAQ-блок с минимум 3 вопросами
[ ] Есть заключение с внутренними ссылками

Источники:
[ ] Минимум 1 гиперссылка на НПА в тексте
[ ] Нет голословных утверждений о суммах/датах без источника

Frontmatter:
[ ] title заполнен, 50–80 символов
[ ] description заполнен, 140–160 символов
[ ] pubDate проставлен (сегодняшняя дата)
[ ] tags: 4–7 штук, строчные
[ ] categories: одна из [ts-piot, kkt, markirovka, egais, zakonodatelstvo]
[ ] draft: true
[ ] seo.keywords: минимум 2 ключа

Стиль:
[ ] Нет эмодзи
[ ] Нет восклицательных знаков (кроме цитат)
[ ] Нет английских вставок в русском тексте
[ ] Активный залог в большинстве предложений
[ ] Числа написаны цифрами (не словами)
```

**При провале:** агент получает список непройденных пунктов → правки только по ним.

**Выход:** статья, соответствующая редполитике

## Стадия 5 — SEO

**Цель:** убедиться, что статья оптимизирована для Яндекса и Google.

**Действия:**

Запустить агент `seo-optimizer` с чеклистом:

**Шлюз 5:**
```
[ ] Целевой ключ присутствует в title (H1)?
[ ] description: 140–160 символов, содержит ключ и ценностный крючок
    (штраф/сумма/срок/пошагово)?
[ ] Целевой ключ присутствует в первом абзаце?
[ ] Целевой ключ присутствует хотя бы в одном H2?
[ ] Внутренних ссылок ≥ 3?
[ ] Год "2026" присутствует в title или description?
```

**При провале:** seo-optimizer получает конкретный список → точечные правки → повтор.
Максимум 2 попытки.

**Выход:** статья с SEO-оптимизированным frontmatter и структурой

## Стадия 6 — КАННИБАЛИЗАЦИЯ

**Цель:** убедиться, что статья не конкурирует с существующими.

**Действия:**

1. Взять seo.keywords новой статьи.
2. Сравнить со всеми seo.keywords опубликованных статей (`draft: false`).
3. Рассчитать пересечение ключей для каждой пары.

**Шлюз 6:**
```
[ ] Пересечение ключей < 50% с любой существующей статьёй?
[ ] Нет другой статьи с тем же целевым запросом в H1?
```

**При провале:** СТОП (автоматические правки невозможны — нужно решение):
```
✗ СТОП: каннибализация с [slug] (пересечение X%).

Варианты:
  A. MERGE — объединить статьи, сделать одну сильную
  B. DIFFERENTIATE — переформулировать угол новой статьи:
     текущий: "[ключ]"
     предлагаю: "[новый угол]"
  C. CANONICAL — новая статья как каноническая, старая → редирект

Выберите вариант (A/B/C):
```

Ожидать ввода пользователя.

**Выход:** подтверждение уникальности или решение пользователя

---

## Стадия 7 — ПУБЛИКАЦИЯ

**Цель:** перевести статью из черновика в опубликованное состояние.

**Действия:**

1. Установить `draft: false` в frontmatter.
2. Убедиться, что `pubDate` = сегодняшняя дата.
3. `git add src/content/blog/SLUG.md`
4. `git commit -m "Публикация: [title] /blog/SLUG/"`
5. `git push`

**Шлюз 7:**
```
[ ] git push прошёл без ошибок?
[ ] draft: false проставлен?
```

**Выход:** статья опубликована, URL: `/blog/SLUG/`

---

## Стадия 8 — REPURPOSE

**Цель:** подготовить контент для дистрибуции в соцсетях.

**Действия:**

Запустить агент `social-media-manager` с опубликованной статьёй.

Сохранить черновики в `src/content/wiki/social/YYYY-MM-DD-slug.md`
со статусом `status: draft`.

**Выход:** черновики для Telegram, VK, Дзен, Email

## Правила выполнения

- Никогда не пропускать стадии, даже если предыдущая прошла быстро.
- При каждом провале шлюза — чётко перечислить что именно не прошло.
- Revision-агент получает только список конкретных проблем, не "улучши статью".
- Максимум 2 retry на любой стадии (кроме каннибализации — там СТОП).
- Если после 2 retry стадия не пройдена — СТОП с подробным объяснением.
- Все промежуточные файлы (brief, social) — коммитить отдельно от статьи.
- Язык всех выводов и правок — только русский.
