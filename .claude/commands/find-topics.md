# /find-topics — найти темы для статей

Анализирует что уже опубликовано, сверяет с контент-планом, ищет незакрытые
запросы через поиск. Возвращает приоритизированный список тем готовых к
написанию.

Без аргумента — обходит все 18 кластеров. С аргументом — только указанный
(ключи перечислены в argument-hint и таблице в `cluster-gaps.md`).

## Алгоритм

### Шаг 1. Аудит текущего контента

1. Прочитать `src/content/wiki/content-plan-2026.md` — кластеры, слаги, статусы.
2. Прочитать все файлы `src/content/blog/*.md` — извлечь slug, title, categories,
   tags, pubDate. Собрать список того, что уже опубликовано (draft: false).
3. Сопоставить с контент-планом: какие строки помечены P0/P1, но ещё не
   опубликованы.

### Шаг 2. Поиск новых запросов

#### 2а. Wordstat weekly diff — главный источник

Прочитать **последний** файл `src/data/wordstat/discoveries/diffs/YYYY-MM-DD.md`
(выбрать самую свежую дату). Он содержит:

- **«🆕 Самые громкие новые запросы за неделю»** — фразы, которых не было в
  прошлой выгрузке. Каждая привязана к `seed` и `cluster`.
- **«📈 Самые крупные приросты»** — фразы, у которых частота выросла ×2+.
- **«По seed-ам»** — детализация по 162 широким семьям ключей.

Алгоритм:
1. Из топ-30 NEW и топ-20 RISING извлечь все фразы.
2. Для каждой нормализовать (lowercase, без кавычек) и проверить — есть ли уже
   статья в блоге/плане. Простое правило: совпадение нормы или подстроки
   с `seo.keywords` любой статьи (`draft: false`) ИЛИ с «Целевым запросом» в
   `content-plan-2026.md`.
3. Что не покрыто → кандидаты в новый план. Кластер берём из `cluster` поля
   соответствующего seed.

Если `diffs/` пуст или последний файл старше 14 дней — отметить и продолжить
только эвристиками (Шаг 2b).

#### 2б. WebSearch для верификации новостной составляющей

Для каждого активного кластера (или указанного в аргументе) выполнить WebSearch.
Поисковые шаблоны зависят от тематики кластера:

**Юридические/налоговые кластеры** (`zakonodatelstvo`, `nalogi`, `kadry`,
`banki`, `otchetnost-edo`):
- `"[тема] 2026" site:nalog.gov.ru OR site:consultant.ru OR site:pravo.gov.ru`
- `"[тема] разъяснение Минфина"`, `"[тема] письмо ФНС"`

**Маркировка и ТС ПИоТ** (`ts-piot`, `markirovka`, `markirovka-2026`):
- `"[категория] маркировка 2026" site:честныйзнак.рф OR site:crpt.ru`
- `"[категория] разрешительный режим 2026"`

**Кассы и ОФД** (`kkt`, `ofd-fn`):
- `"[модель] обзор" OR "[модель] характеристики"` для подбора моделей
- `"ФФД 1.2 [тема] 2026"`, `"ОФД [тема] 2026 site:nalog.gov.ru"`

**Алкоголь** (`egais`, `horeca`):
- `"[тема] ЕГАИС 2026" site:fsrar.gov.ru OR site:egais.ru`
- `"[тема] лицензия на алкоголь 2026"`

**Маркетплейсы** (`marketplace`):
- `"WB [тема] 2026" site:seller.wildberries.ru` / `site:seller.ozon.ru`
- `"[тема] комиссия маркетплейс 2026"`

**Меркурий** (`merkuriy`):
- `"[тема] Меркурий 2026" site:vetrf.ru OR site:fsvps.gov.ru`
- `"ВСД [тема]"`

**Стартующий бизнес и услуги** (`start`, `uslugi`):
- `"как открыть [тип бизнеса] 2026"`, `"патент [деятельность] 2026"`
- `"первая касса [деятельность]"`

Общие правила для любого кластера:
- Информационные запросы с высоким интентом: `"[тема] как"`, `"[тема] что такое"`,
  `"[тема] штраф"`, `"[тема] сроки"`, `"[тема] инструкция"`.
- Вопросы из «Люди также спрашивают» Яндекса по ключевым темам.
- Проверить, не появились ли новые сроки или изменения законодательства
  после даты последней публикации в кластере.

### Шаг 3. Проверка на каннибализацию

Для каждой найденной темы проверить — нет ли уже похожей статьи в блоге.
Двухуровневая проверка:

1. **Лексическая (быстро):** пересечение нормализованных целевых ключей
   с `seo.keywords` существующих статей. Порог > 50% → флаг.
2. **Семантическая (точно):** если есть `src/data/audit/similarity.json`
   (генерируется workflow `embeddings-monthly.yml`), для самых близких по
   ключам статей дополнительно проверяем cosine similarity к новой теме:
   - При наличии embedding для слова-кандидата (или ad-hoc эмбеддинг
     через тот же провайдер) — пары с cos ≥ 0.85 считаются прямыми
     дублями, 0.75–0.85 — углами одной темы (предложить расширение).
   - Если embeddings отсутствуют — работаем только по лексическому уровню.

Если найден дубль (любого уровня) — предложить **обновление**, а не новую
статью.

### Шаг 4. Приоритизация

Перед ранжированием прочитать `src/data/wordstat/keys.json` — кеш частот из
Wordstat API. Структура и поля описаны в `docs/wordstat.md`. Для каждой
найденной темы:

- Нормализовать целевой запрос (lowercase, без кавычек, схлопнуть пробелы).
- Найти в `keys[<норм>]` поля:
  - `shows` — частота точно этой фразы за последний месяц.
  - `topShows` — максимум среди популярных вариаций (`related[].count`).
  - `topPhrase` — самая популярная вариация.
  - `trend` — `up`/`down`/`stable`/`unknown`.
- **Реальный спрос на тему = `max(shows, topShows)`**. Если `topShows`
  больше `shows` в 3+ раза — целевой запрос сформулирован слишком узко,
  предложить переформулировку через `topPhrase` в комментарии к строке.
- Если ключа нет — добавить его в `seo.keywords` будущей статьи (попадёт
  в кеш на следующем weekly-обновлении) и работать как раньше.
- Дополнительно подсмотреть `related` у близких ключей — это готовый список
  смежных формулировок, который Яндекс уже знает; источник идей для новых
  тем-сателлитов.

Ранжировать темы по:
1. **Срочность** — есть конкретный дедлайн / вступающий в силу закон.
2. **Поисковый спрос** — `max(shows, topShows)` из `keys.json`. Порядок:
   - `≥ 2000` или `trend = up` → P0
   - `500–2000` → P1
   - `< 500` → P2 (если только нет срочности)
3. **Пробел в кластере** — нет покрытия ни одной статьёй.
4. **Сложность написания** — простые информационные статьи в приоритете над
   сложными аналитическими.

Если `lastFullUpdate` в `keys.json` старше 14 дней — отметить «частоты
устарели, цифры ориентировочные» в выводе и предложить запустить workflow
**Wordstat Weekly Refresh**.

## Формат вывода

Таблица Markdown, готовая к вставке в `content-plan-2026.md`:

```markdown
| # | Слаг | Тема | Целевой запрос | Приоритет | Wordstat (точно / вокруг / trend) | Обоснование |
|---|------|------|---------------|-----------|-----------------------------------|-------------|
| 1 | slug-temy | Заголовок темы | ключевой запрос | P0 | 2400 / 5800 / up | Срок вступления в силу: ДД.ММ.ГГГГ |
```

Если ключа нет в кеше — в колонке Wordstat поставить `—`. Если `topShows`
сильно больше `shows` — добавить в Обоснование заметку
«переформулировать на «<topPhrase>»».

Плюс отдельный блок «Рекомендую обновить» — статьи старше 6 месяцев по теме,
у которых появились новые НПА или изменились сроки.

## Ограничения

- Не предлагать темы, которые дублируют существующие статьи.
- Не предлагать коммерческие темы («лучший сервис», «топ-10 касс») — только
  информационный интент.
- Минимум 5, максимум 15 новых тем за один запуск.
- Если в `content-plan-2026.md` появился новый кластер — добавить его ключ в
  argument-hint и в эталонные запросы `cluster-gaps.md`.
