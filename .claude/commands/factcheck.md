# /factcheck — проверка фактов одной статьи

Извлекает claims из статьи (даты, суммы, упоминания НПА, ссылки) и сверяет
каждый с актуальным первоисточником. Возвращает JSON-отчёт со статусами и
рекомендации по правкам.

Запускается по slug: `/factcheck 2026-01-15-chto-takoe-ts-piot`.

## Алгоритм

### Шаг 1 — Извлечь claims

```bash
node scripts/factcheck/extract-claims.mjs <slug>
```

Скрипт создаёт `src/data/factcheck/claims/<slug>.json` с массивом claims:
DATE_TEXT, DATE_DMY, DATE_YEAR, DATE_RANGE, MONEY, NPA_KOAP, NPA_FZ, LINK,
PERCENT. Каждый claim имеет `raw`, `context` (±80 символов), `line`,
`offset`.

### Шаг 2 — Категоризировать claims

Прочитать `src/data/factcheck/sources.json` — там маппинг типов claims на
канонические первоисточники + список заблокированных доменов (для них
работаем через WebSearch, а не WebFetch).

Решить для каждого claim:
- **Skip** — не нуждается в проверке (внутренняя ссылка на наш блог,
  ссылка на якорь и т.д.).
- **Check** — нужна проверка через WebFetch.
- **Search** — первоисточник в `blocked.domains` → используем WebSearch.

### Шаг 3 — Проверить каждый claim

Для каждого «Check» — `WebFetch` к первоисточнику, спросить актуальное
значение для контекста claim.

Для каждого «Search» — `WebSearch` с конкретным запросом, ориентирующимся
на claim. Шаблоны:
- **NPA_KOAP**: `"статья X.Y КоАП РФ 2026 [контекст из claim, например 'штраф ИП']"`
- **DATE_TEXT**: `"[тема статьи] [дата] постановление правительства"`
- **MONEY**: запросить вместе с упомянутой статьёй КоАП.

### Шаг 4 — Сравнить и записать результат

Для каждого проверенного claim — записать в массив с полями:
- `status` — `match` / `mismatch` / `uncertain` / `missing`
- `severity` — `critical` / `moderate` / `minor`
- `confidence` — 0.0–1.0
- `expectedValue` — что говорит первоисточник
- `explanation` — суть расхождения
- `sources` — массив URL-ов, на которых нашли подтверждение
- `action` — `rewrite-lede` / `rewrite-bullet` / `expand-bullet` / `add-references` / `keep`
- `actionDetail` — конкретный текст / правка

Записать всё в `src/data/factcheck/results/<slug>.json`. Шаблон см. в
`src/data/factcheck/results/2026-01-15-chto-takoe-ts-piot.json`.

### Шаг 5 — Создать маркер фактчека

Записать пустой файл `.claude/factchecked/<slug>` с датой проверки —
используется `/maintain-content` и `/release-article` как признак, что
статья прошла проверку.

### Шаг 6 — Вывести отчёт

В чат выдать summary:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✔️ /factcheck <slug> — отчёт YYYY-MM-DD
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Проверено N claims из M.

🔴 Критические расхождения (X):
  • [тип] «raw» — суть проблемы → действие

🟡 Средние (Y):
  • ...

🟢 Совпадает (Z)

📋 Рекомендации:
  1. ...
  2. ...

Полный JSON: src/data/factcheck/results/<slug>.json
```

## Когда статус — `mismatch` критической тяжести

Если найдено ≥ 1 `critical mismatch`:
1. **Не публиковать без правки.** Если у статьи `draft: false`, поставить
   `draft: true` обратно до исправления.
2. Предложить пользователю набор правок согласно `action`/`actionDetail`.
3. После применения правок — повторно запустить `/factcheck` (без аргумента
   или с тем же slug).

## Маркетинговый баланс — применяй редполитику

**Источник истины: `docs/editorial-policy.md`.** Не каждое расхождение
правится под букву первоисточника — там есть три класса правил.

Сценарий применения политики **в момент проверки**:

1. Прочитать `docs/editorial-policy.md` (правила классов A/B/C).
2. Для каждого расхождения подобрать правило по таблице «Что делает
   скилл автоматически».
3. Применить соответствующее `action`:
   - **Класс A** (`A1` суммы, `A2` номера НПА, `A3` техника, `A4` названия,
     `A5` юр. термины) → `severity: critical`, `action: rewrite-*`,
     откатить в `draft: true` если не исправлено.
   - **Класс B** (`B1` дедлайны, `B2` аудитория, `B3` тон, `B4` CTA,
     `B5` срочность) → `severity: moderate`,
     `action: add-disclaimer` / `keep`. Сохранить формулировку автора +
     добавить контекстный блок.
   - **Класс C** (`C1` внутренние ссылки, `C2` общие утверждения,
     `C3` якоря) → `status: skip`, в отчёт не пишем.

4. Если ситуация не подходит ни под одно правило — в отчёте поставить
   `status: needs-decision` и оставить редактору на ручное решение.

5. Если в процессе обнаружено типовое расхождение, не покрытое политикой
   — предложить пользователю добавить новое правило в
   `docs/editorial-policy.md`.

**Не изобретай решения на лету.** Если нет правила — лучше
`needs-decision`, чем самостоятельная интерпретация. Политика прирастает
явно, через коммиты.

## Обязательные точки проверки (типовые пробелы)

Бенчмарк против Контура вскрыл повторяющиеся промахи. Проверяй их всегда,
даже если claim не помечен как спорный:

1. **Штрафы — субъект, не только сумма.** Частая ошибка — приписать ИП тир
   юрлица (или наоборот). Проверь раздельно: ИП / должностное лицо / юрлицо.
   Обязательно проверь, применимо ли **смягчение для МСП (ст. 4.1.2 КоАП РФ —
   вдвое меньше)** и упомянуто ли оно. Проверь наличие **уголовной статьи**
   (например ст. 171.1 УК РФ), если тема о немаркированном обороте.
2. **Символики кодов.** Data Matrix (маркировка) и PDF417 vs Data Matrix на
   акцизных/федеральных спецмарках — форматы меняли. Проверяй актуальную
   символику на ТЕКУЩИХ марках, не по памяти.
3. **Версии оборудования и токенов.** Рутокен ЭЦП 2.0/3.0, JaCarta и т. п. —
   версии устаревают. Проверяй актуальную.
4. **Обязанности по ролям.** Одна ли электронная подпись на несколько
   госсистем или для ЕГАИС нужна отдельная КЭП — проверяй, не упрощай.
5. **Персональные данные (после реформы 2025).** Локализация базы в РФ, сроки
   уведомления об утечке (24 ч / 72 ч), согласие отдельным документом, тиры и
   срок уведомления в РКН, оборотные штрафы — все недавно менялись.
6. **Эскалация при противоречии.** Если наш тезис расходится с действующим
   официальным источником или сильным конкурентом — `severity: critical`,
   даже если цифра выглядит правдоподобно.

## Ограничения

- WebFetch блокируется на consultant.ru, pravo.gov.ru, yandex.* — используем
  WebSearch. Ответ берётся из сниппетов и заголовков выдачи.
- Скрипт извлечения работает по regex — может пропустить нестандартные
  упоминания (например, «по статье пятнадцать двенадцать»). Это нормально:
  фактчек — не замена редактору, а первый фильтр.
- Один прогон ≈ 10 WebSearch + 0–5 WebFetch. Для batch (#39 в backlog) —
  параллельный subagent dispatch.
