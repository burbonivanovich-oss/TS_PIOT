# /maintain-content — обслуживание контента

**Процесс 4 из 4.** Запускается раз в квартал или при изменении законодательства.
Результат — актуализированные статьи с обновлёнными датами и фактами.

Вызов без аргумента — аудит всех опубликованных статей.
С аргументом — только одна статья: `/maintain-content 2026-01-15-chto-takoe-ts-piot`.

---

## Шаг 1 — Инвентаризация

### 1а. Maintain-queue (приоритет)

Если существует `src/content/wiki/maintain-queue.md` — прочитать его в
первую очередь. Это явные кандидаты на обновление из `/monitor-rss`
(поток `guidelines`): методички, разъяснения ведомств, письма ФНС
без дедлайна, которые попадают в существующие статьи как контекст,
а не в `/plan-content` как новые темы.

Формат записи в `maintain-queue.md`:
```
- <slug> — <что добавить> (источник: <URL>)
```

Эти статьи идут в очередь аудита **первыми** с пометкой «из RSS».

### 1б. Авто-инвентаризация по маркерам

Прочитать frontmatter всех файлов в `src/content/blog/` с `draft: false`.
Для каждой статьи также прочитать маркер `.claude/factchecked/<slug>` —
он содержит ISO-дату последнего фактчека.

Построить таблицу:

| Slug | updatedDate | reviewDate | factchecked | reviewDate≤today | factchecked>90d |
|---|---|---|---|---|---|

В очередь аудита попадают статьи, у которых выполнено **хотя бы одно**:
- статья есть в `maintain-queue.md`;
- `reviewDate` ≤ сегодня;
- маркер `.claude/factchecked/<slug>` старше 90 дней (квартал) **или
  отсутствует**;
- в `src/data/wordstat/keys.json` для одного из `seo.keywords` поле
  `trend = "up"` или `shows` вырос ×2+ к среднему за 6 месяцев.

Сортировка очереди:
1. Сначала из `maintain-queue.md` (явные кандидаты — конкретная подсказка
   из RSS).
2. Потом без маркера фактчека.
3. Потом по возрасту маркера (старые впереди).
4. Потом по `reviewDate`.

### 1в. После обработки — удалить запись из maintain-queue.md

Когда статья из `maintain-queue.md` пройдена и обновлена — вычеркнуть
её из файла (или удалить строку). Коммитом
`chore(maintain-queue): удалить <slug> — обработано`.

## Шаг 2 — Аудит каждой статьи

Для каждой статьи из очереди выполнить:

### 2а. Фактчек

Запустить `/factcheck <slug>` — наш собственный скилл (см. `docs/factcheck.md`).
Скрипт извлекает claims, скилл сверяет через WebSearch, применяет
редполитику (`docs/editorial-policy.md`). Результат: `src/data/factcheck/results/<slug>.json`
+ обновлённый маркер `.claude/factchecked/<slug>`.

Из отчёта берём:
- `summary.criticalIssues` > 0 → статья **обязательно** идёт в «ОБНОВИТЬ».
- `summary.overallStatus` = `match` / `good-with-minor-addition` → можно
  оставить как «АКТУАЛЬНА» при условии что остальные критерии тоже OK.
- `summary.overallStatus` = `needs-rewrite` → «ПЕРЕПИСАТЬ».

### 2а-bis. Динамика спроса (Wordstat)

Прочитать `src/data/wordstat/keys.json`. Для каждого ключа из `seo.keywords`
статьи (нормализованного — lowercase, без кавычек):

1. Найти запись `keys[<норм>]`. Если нет — пропустить.
2. Из `history` (массив `[{date: "YYYY-MM", count: N}, ...]`) вычислить:
   - `recent3` = среднее по последним 3 месяцам;
   - `baseline6` = среднее по 6 месяцам, предшествующим recent3 (т.е.
     история длиной 9 мес: первые 6 — baseline, последние 3 — recent).
   - Если истории < 9 точек — используем `trend` поле напрямую
     (`up`/`down`/`stable`).
3. Классифицировать:
   - `recent3 / baseline6 ≥ 2` → **📈 спрос растёт** — критичный сигнал
     «срочно обновить, расширить, добавить FAQ». В отчёте: severity high.
   - `recent3 / baseline6 ≤ 0.5` → **📉 спрос падает** — кандидат на
     объединение с более крупной статьёй или архивацию.
   - Между → стабильно, не упоминаем.
4. Дополнительно — посмотреть `topShows` и `topPhrase`. Если `topShows ≥
   3 × shows` целевого ключа — формулировка `seo.keywords` слишком узкая.
   В отчёте предложить переформулировку на `topPhrase`.

Эти сигналы попадают в отчёт Шага 3 рядом с balls и factcheck-статусом.

### 2б. Оценка по 10-балльной шкале

| Критерий | Балл |
|---|---|
| Объём ≥ 1500 слов | 1 |
| Структура: 5+ H2, FAQ, заключение | 1 |
| Внутренних ссылок ≥ 3 | 1 |
| Нет устаревших фактов/дат | 2 |
| SEO: ключ в title + абзаце + H2 + description OK | 2 |
| Стиль: нет паразитов, активный залог | 1 |
| Hero-изображение есть | 1 |
| Внутренние ссылки ведут на существующие страницы | 1 |

### 2в. Решение: что делать со статьёй

| Балл | Решение | Критерий |
|---|---|---|
| 8–10 + факты актуальны | **АКТУАЛЬНА** — обновить reviewDate | — |
| 5–7 или 1–2 устаревших факта | **ОБНОВИТЬ** — правки по списку | Та же целевая аудитория и интент |
| < 5 или кардинально изменился закон | **ПЕРЕПИСАТЬ** — как новую через `/create-article` | Старая статья → `draft: true` |
| Дублирует более сильную статью | **ОБЪЕДИНИТЬ** — merge + canonical redirect | Каннибализация > 70% |
| Тема потеряла актуальность | **АРХИВИРОВАТЬ** — `draft: true` без замены | — |

**Правило обновить vs новая статья:**
- Обновляем, если статья уже ранжируется по целевому ключу (даже в топ-30).
- Пишем новую, если статья не ранжируется + прошло 3+ месяца после публикации.
- Переписываем, если изменился сам закон (не цифры, а суть требований).

## Шаг 4 — Выполнение

### Для ОБНОВИТЬ:

1. Внести правки по `recommendations` из `src/data/factcheck/results/<slug>.json`
   (только конкретные пункты, не «улучши статью»).
2. Добавить/обновить `updatedDate: YYYY-MM-DD`.
3. Обновить `reviewDate` на +3 месяца (квартал), не +6 — синхрон с
   циклом re-factcheck.
4. Перезапустить `/factcheck <slug>` чтобы маркер обновился новой датой.
5. Закоммитить: `update(content): актуализация [slug] — [что изменено]`

### Для ПЕРЕПИСАТЬ:

Запустить `/create-article "<тема>"` с новым углом.
Старую статью перевести в `draft: true` после публикации новой.

### Для АКТУАЛЬНЫ:

1. Обновить только `reviewDate` в frontmatter на +3 месяца.
2. Перезапустить `/factcheck <slug>` — даже если статья актуальна, маркер
   должен обновиться датой, чтобы следующая инвентаризация не выловила её
   снова через 90 дней.
3. Закоммитить: `chore(content): обновить reviewDate — [slug]`.

### Для АРХИВИРОВАТЬ:

Установить `draft: true`. Закоммитить:
`chore(content): архивировать [slug]`

## Правила

- Никогда не удалять статьи — только `draft: true`.
- Не переписывать без подтверждения пользователя.
- При обновлении менять только то, что в списке правок — не рефакторить стиль.
- Язык — только русский.
