# /maintain-content — обслуживание контента

**Процесс 4 из 4.** Запускается раз в квартал или при изменении законодательства.
Результат — актуализированные статьи с обновлёнными датами и фактами.

Вызов без аргумента — аудит всех опубликованных статей.
С аргументом — только одна статья: `/maintain-content 2026-01-15-chto-takoe-ts-piot`.

---

## Шаг 1 — Инвентаризация

Прочитать frontmatter всех файлов в `src/content/blog/` с `draft: false`.

Построить таблицу:

| Slug | updatedDate | reviewDate | Прошёл reviewDate? |
|---|---|---|---|

Статьи, у которых `reviewDate` ≤ сегодня — приоритетные для аудита.

---

## Шаг 2 — Аудит каждой статьи

Для каждой статьи из очереди (или только указанной) выполнить:

### 2а. Проверка дат и норм

Прочитать статью. Найти все упоминания:
- конкретных дат (дедлайны, сроки)
- сумм штрафов и санкций
- номеров НПА и постановлений
- названий программ и реестров

Сверить с актуальным состоянием на дату аудита (использовать WebSearch если доступен).

Пометить найденные расхождения: ✗ устарело / ⚠ требует проверки / ✅ актуально.

### 2а-bis. Динамика спроса (Wordstat)

Прочитать `src/data/wordstat/keys.json`. Для целевого запроса статьи и каждого
ключа из `seo.keywords`:

- Если `shows` вырос ×2 или больше относительно средней за первые 3 месяца
  истории — добавить пометку **`📈 спрос растёт`** в раздел статьи. Это
  сильный сигнал «срочно обновить и расширить».
- Если `shows` упал в 2 раза — пометка **`📉 спрос падает`**, кандидат на
  объединение или архивацию.
- Если ключа нет в кеше — пропускаем (Wordstat обновится на следующем weekly).

### 2б. Оценка по 10-балльной шкале

| Критерий | Балл |
|---|---|
| Объём ≥ 1500 слов | 1 |
| Структура: 5+ H2, FAQ, заключение | 1 |
| Внутренних ссылок ≥ 3 | 1 |
| Нет устаревших фактов/дат | 2 |
| SEO: ключ в title + абзаце + H2 + description OK | 2 |
| Стиль: нет паразитов, активный залог | 1 |
| Hero-изображение есть | 1 |
| Внутренние ссылки ведут на существующие страницы | 1 |

### 2в. Решение: что делать со статьёй

| Балл | Решение | Критерий |
|---|---|---|
| 8–10 + факты актуальны | **АКТУАЛЬНА** — обновить reviewDate | — |
| 5–7 или 1–2 устаревших факта | **ОБНОВИТЬ** — правки по списку | Та же целевая аудитория и интент |
| < 5 или кардинально изменился закон | **ПЕРЕПИСАТЬ** — как новую через `/create-article` | Старая статья → `draft: true` |
| Дублирует более сильную статью | **ОБЪЕДИНИТЬ** — merge + canonical redirect | Каннибализация > 70% |
| Тема потеряла актуальность | **АРХИВИРОВАТЬ** — `draft: true` без замены | — |

**Правило обновить vs новая статья:**
- Обновляем, если статья уже ранжируется по целевому ключу (даже в топ-30).
- Пишем новую, если статья не ранжируется + прошло 3+ месяца после публикации.
- Переписываем, если изменился сам закон (не цифры, а суть требований).

## Шаг 4 — Выполнение

### Для ОБНОВИТЬ:

1. Внести правки по списку (только конкретные пункты, не «улучши статью»).
2. Добавить/обновить `updatedDate: YYYY-MM-DD`.
3. Обновить `reviewDate` на +6 месяцев.
4. Если изменились ключевые факты — повторить factcheck:
   `/blog factcheck src/content/blog/SLUG.md`
5. Закоммитить: `update(content): актуализация [slug] — [что изменено]`

### Для ПЕРЕПИСАТЬ:

Запустить `/create-article "<тема>"` с новым углом.
Старую статью перевести в `draft: true` после публикации новой.

### Для АКТУАЛЬНЫ:

Обновить только `reviewDate` в frontmatter. Закоммитить:
`chore(content): обновить reviewDate — [slug]`

### Для АРХИВИРОВАТЬ:

Установить `draft: true`. Закоммитить:
`chore(content): архивировать [slug]`

## Правила

- Никогда не удалять статьи — только `draft: true`.
- Не переписывать без подтверждения пользователя.
- При обновлении менять только то, что в списке правок — не рефакторить стиль.
- Язык — только русский.
