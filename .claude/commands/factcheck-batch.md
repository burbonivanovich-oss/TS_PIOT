# /factcheck-batch — пакетный фактчек

Запускает `/factcheck` логику для 8–16 статей одновременно через
параллельный subagent dispatch. За одну сессию (~30 минут) закрывает
весь корпус: при backfill 2026-05-19 — 75 статей за 18 батчей.

Вызов:
- `/factcheck-batch` — авто-выбор 8 статей риск-зоны без маркера.
- `/factcheck-batch --count 16` — 16 статей (4 батча × 4 параллельно).
- `/factcheck-batch --filter markirovka` — все непроверенные маркировки.
- `/factcheck-batch --filter risk-zone` — налоги/штрафы/маркировка.

---

## Алгоритм

### Шаг 1 — Отбор кандидатов

```js
const factchecked = new Set(fs.readdirSync('.claude/factchecked/'));
const candidates = readBlogDir()
  .filter(slug => !factchecked.has(slug))
  .filter(slug => isPublished(slug));
```

Если `--filter risk-zone` — фильтр по регекспу
`shtraf|штраф|nalog|налог|nds|usn|patent|psn|kkt|ts-piot|markirovka|54-fz|samozanyatyj`
(те же категории что в backfill сессии 2026-05-19).

Если `--filter <topic>` — фильтр по части slug'а или категории.

Лимит — 16 статей за прогон (4 батча × 4 параллельно). По умолчанию 8.

### Шаг 2 — Группировка в батчи

Разбить кандидатов на батчи по 4 статьи, **группируя по теме** для
переиспользования whitelist'а агентом:

- markirovka — отдельный батч, общий контекст про ПП маркировки
- ккт / ОФД / ФН — отдельный батч, контекст 54-ФЗ
- ЕГАИС / Меркурий — отдельный батч, контекст 171-ФЗ и Меркурия
- налоги (УСН/НДС/ПСН) — отдельный батч, контекст ФЗ-425 и дефляторов

Тематика → шаблон промпта для агента (см. ниже «Шаблоны батчей»).

### Шаг 3 — Параллельный dispatch

Запустить 2–4 subagent-а одновременно (один Agent tool-call с несколькими
параметрами). Каждый агент:

1. Получает свой список из 4 статей
2. Для каждой запускает `node scripts/factcheck/extract-claims.mjs <slug>`
3. Каждый claim → WebSearch к первоисточнику (`src/data/factcheck/sources.json`)
4. Если class C — точечная правка статьи, `updatedDate: 2026-05-19`,
   `draft: false` сохраняется
5. Записывает `src/data/factcheck/results/<slug>.json`
6. Создаёт `.claude/factchecked/<slug>` с датой

Тип агента: `general-purpose`.

`run_in_background: true` — пользователь не ждёт активно, получает
notification по мере завершения каждого батча.

### Шаг 4 — Сбор отчёта

По мере завершения батчей:
- Накапливать список фейковых НПА (особенно в категории НПА_PP_NUMBERED)
- Накапливать системные ошибки (повторяющиеся в нескольких статьях)
- Закоммитить результаты + маркеры

Финальный коммит с описанием:
- сколько проверено
- сколько critical fixes
- какие фейковые НПА найдены (для пополнения whitelist'а)
- новые системные паттерны

### Шаг 5 — Пополнение whitelist (если что-то нашли)

Если агент нашёл реальную норму, которой нет в
`src/data/factcheck/sources.json.npaWhitelist`:
1. Открыть sources.json
2. Добавить в `npaWhitelist.{fz|pp|prikaz|rasporyazhenie}.<number>`:
   `{ "title": "...", "date": "YYYY-MM-DD", "topic": "..." }`
3. Закоммитить отдельным `chore(#31): whitelist пополнен`

## Тайминг

| Параметр | Значение |
|---|---|
| Один батч × 4 статьи | ~5–10 минут реального времени |
| 4 батча параллельно | ~10 минут реального времени |
| 16 статей за прогон | ~10–15 минут |
| Полный backfill 80 статей | ~50 минут (5 прогонов) |

Сравнение с последовательным `/factcheck` (по одной статье через скилл):
80 × 5 минут = 6.5 часов. Параллельный dispatch даёт **×6 ускорение**
без потери качества (см. `docs/session-2026-05-19-postmortem.md`).

---

## Правила

- Не запускать > 4 параллельных батчей одновременно — WebSearch rate-limit.
- Не править статьи без critical. Class B (moderate) — только в отчёт,
  не редактировать.
- Не публиковать (`draft: true` → `false`) — только обновлять
  `updatedDate` при исправлении.
- При обнаружении фейкового НПА — записать в коммит-сообщение для
  будущей history (см. `docs/factcheck-history.md`).
- Не запускать `git commit/push` из агента — это делает оркестратор.
- Язык отчётов — только русский.
