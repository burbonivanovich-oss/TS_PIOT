# /create-article — создание статьи

**Процесс 2 из 4.** Запускается по теме из редкалендаря.
Результат — статья в `draft: true`, прошедшая все проверки, с hero-изображением.

Вызов: `/create-article "штраф за работу без ТС ПИоТ в 2026"`

Для публикации — `/release-article <slug>`.
Для быстрого черновика без шлюзов — `/new-post`.

---

## Статус

```
▶ /create-article "<тема>"
━━━━━━━━━━━━━━━━━━━━━━━━━━━
Стадия 1/9: DISCOVERY
```

---

## Стадия 1 — DISCOVERY

1. Взять тему из аргумента или из нашего плана `src/data/editorial-plan.json`;
   уточнить целевой запрос.

2. Проверить дублирование по карте покрытия: `src/data/content-plan.json`
   (что уже покрыто) + `src/data/kontur-links.json` (статьи Контура) +
   frontmatter всех `src/content/blog/*.md`. Пересечение по смыслу > 70% —
   СТОП, предложить варианты (обновление вместо дубля).

3. **Определить, попадает ли тема в категорию с НПА:**
   - `ts-piot` → ПП № 1944, 54-ФЗ
   - `markirovka` → отраслевые ПП Правительства, ст. 15.12 КоАП, ст. 171.1 УК
   - `zakonodatelstvo` → НК РФ, ТК РФ, КоАП, ФЗ
   - `egais` → ст. 14.16/14.17/14.19 КоАП, 171-ФЗ
   - `kkt` → 54-ФЗ, ст. 14.5 КоАП

   Если категория попадает в этот список — **research-specialist обязателен.**
   Без research-brief writer не запускается. Это страховка от галлюцинаций
   номеров НПА (за прошлые сессии выявлено 3 фейковых ПП).

4. Запустить агент `research-specialist`. Сохранить brief:
   `.claude/research/YYYY-MM-DD-slug-brief.md`. Bief должен содержать
   frontmatter с массивом `sources[]`, каждый элемент с `verified: true`
   (агент физически проверяет норму через WebFetch/WebSearch).

**Шлюз 1:**
```
[ ] Тема не дублирует существующие статьи?
[ ] Категория с НПА → research-brief обязателен; иначе пропуск разрешён?
[ ] Brief содержит ≥ 3 источника с НПА/датами (тир + дата вступления)?
[ ] Все sources[] в brief frontmatter имеют verified: true?
[ ] Есть целевой поисковый запрос?
[ ] Проставлены product / segment / contentType (из таксономии редплана)?
[ ] Бенчмарк-паас: найдена статья Контура по теме (konturArticle) с пробелами,
    либо отмечено, что статьи нет?
[ ] Закрыты обязательные точки проверки (штрафы по субъектам + МСП + УК,
    символики кодов, версии токенов) — где применимо?
```

## Стадия 2 — WRITE

Запустить агент `content-writer` с research brief.

Требования:
- 1200–2500 слов, структура: лид → 5–7 H2 → FAQ (≥ 4 вопроса) → заключение
- ≥ 3 ссылки на статьи Контура (реальные URL из src/data/kontur-links.json
  или заглушки), ≥ 1 ссылка на НПА. Ссылки на наш /blog/ не используем.
- Frontmatter по шаблону из CLAUDE.md, `draft: true`
- Оглавление «В этой статье» после лида (список H2)
- `author` из `src/data/authors.json` (реальный, не дефолт) + `updatedDate`
- `faq[]` во frontmatter продублирован из FAQ-блока (FAQPage-разметка)
- Голос по калибровке из content-writer (так/не так): без казёнщины и пассива
- CPA-врезка по `product` из брифа (market/ofd/elba)
- Штрафные блоки полные: ИП / должностное / юрлицо + смягчение для МСП
  (ст. 4.1.2 КоАП) + уголовная статья, где применима

**Образец «как надо».** Ориентир для writer'а — статья
`src/content/blog/2026-05-01-smart-terminal-vs-fiskalnyj-registrator.md`:
конкретика и цифры, сценарии «для кого подходит», сравнительная таблица,
расчёт стоимости, термины в скобках при первом упоминании, сильный лид,
FAQ и заключение с внутренними ссылками. К такому уровню тянем каждую статью.

Сохранить: `src/content/blog/YYYY-MM-DD-slug.md`

**Шлюз 2 — техминимум:**
```
[ ] Файл создан и парсится без ошибок?
[ ] Frontmatter полный (title, description, pubDate, tags, categories, seo.keywords)?
[ ] Тело статьи ≥ 1000 слов?
```

**Шлюз 2 — «Хорошо написано» (7 критериев, да/нет по каждому):**
```
[ ] 1. Конкретика вместо воды — цифры, характеристики, названия, цены, сроки
       (не «устройство удобное», а «батарея 8–12 часов»)
[ ] 2. Читателю понятно, что делать — сценарии «для кого/когда», шаги, готовый выбор
[ ] 3. Разбор через примеры и ситуации — правило подкреплено живым кейсом бизнеса
[ ] 4. Термины объясняются сразу — сложное слово раскрыто в скобках при первом упоминании
[ ] 5. Сравнения и расчёты, где уместно — там, где выбирают, есть таблица/расчёт
[ ] 6. Сильное начало — в первом абзаце сразу проблема и польза, без разгона
[ ] 7. Полезное завершение — короткий вывод + куда идти дальше (ссылки на наши статьи)
```

Плюс автоматическая проверка на нейросетевой/шаблонный текст:
```bash
node scripts/check-ai-markers.mjs src/content/blog/YYYY-MM-DD-slug.md
```

**Правило прохождения шлюза 2:**
- Техминимум выполнен **и** выполнено **≥ 6 из 7** критериев «Хорошо написано»
  **и** `check-ai` даёт скор ≤ 5 (нет красного) → зачёт, идём дальше.
- **5 и меньше** критериев или `check-ai` ≥ 6 → на переписывание
  (максимум 2 попытки, как на других стадиях; revision-агенту передать
  только список непройденных пунктов).

## Стадия 2а — ПЕРЕЛИНКОВКА НА КОНТУР

Перелинковка ведёт на статьи Контура (kontur.ru), НЕ на наш `/blog/` —
проект работает во благо Контура.

1. Для каждой смежной темы найди статью Контура в базе
   `src/data/kontur-links.json` по совпадению заголовка. Поставь **реальный
   URL** в markdown-ссылку с описательным якорным текстом.
2. Где подходящей статьи в базе нет — оставь текстовую заглушку:
   `тема (нужна ссылка на статью Контура про «тема»)`. Не выдумывай URL.
3. Минимум 3 ссылки на Контур (реальные URL или заглушки) в теле статьи.

**Шлюз 2а:**
```
[ ] У статьи ≥ 3 ссылок на Контур (реальные URL из базы или заглушки)?
[ ] Ни одной ссылки на наш /blog/ или /category/ не осталось?
[ ] Все реальные URL присутствуют в src/data/kontur-links.json (не выдуманы)?
```

## Стадия 2б — CALLOUTS

Прочитать статью. Найти 1–2 факта для врезок:
- `callout--penalty` — если есть конкретные суммы штрафов
- `callout--date` — если есть ключевая дата вступления в силу
- `callout--warning` — если есть критическое требование
- `callout--tip` — если есть пошаговый раздел

Вставить через `<div class="callout callout--TYPE">...</div>`.
Контент — только из текста статьи, не придумывать.

**Шлюз 2б:** не более 2 врезок, факты только из текста.

---

## Стадия 3 — FACTCHECK

1. Прогнать NPA-аудит: `node scripts/factcheck/audit-npa-references.mjs --strict`.
   Если в статье есть упоминания ПП/Приказов/ФЗ, которых нет ни в whitelist
   `src/data/factcheck/sources.json`, ни на pravo.gov.ru — это критическое
   расхождение. Прежде чем запускать `/factcheck`, исправь номера или
   расширь whitelist.

2. Запустить `/factcheck <slug>` (наш собственный стек, не плагин claude-blog).
   Скилл извлекает claims, сверяет с первоисточниками, применяет редполитику
   из `docs/editorial-policy.md` (классы A/B/C). Подробности —
   `docs/factcheck.md`.

**Шлюз 3:**
```
[ ] audit-npa-references.mjs --strict зелёный?
[ ] Нет claims с confidence < 0.7 без пометки [нужна проверка]?
[ ] Все даты сроков подтверждены источником?
[ ] Суммы штрафов соответствуют актуальной редакции КоАП?
[ ] Штраф привязан к верному субъекту (ИП / должностное / юрлицо) + учтено
    смягчение для МСП (ст. 4.1.2 КоАП)?
[ ] Символики кодов и версии токенов актуальны (Data Matrix/PDF417, Рутокен 2.0/3.0)?
[ ] Все ПП/Приказы/ФЗ в статье — из npaWhitelist или brief.sources[]?
```

При провале — таблица слабых claims → правки → повтор. Максимум 2 попытки.

## Стадия 4 — РЕДПОЛИТИКА

Сверить статью с CLAUDE.md (разделы «Правила контента», «Стиль речи»).

**Шлюз 4 (ключевые точки отказа):**
```
Объём:
[ ] ≥ 800 слов (сателлит) или ≥ 1500 (опорная)

Структура:
[ ] ≥ 4 H2, FAQ ≥ 3 вопросов, заключение с внутренними ссылками
[ ] H2-заголовки — тема, не вопрос; форма «Каков/Что такое» — запрещена

Frontmatter:
[ ] title: 50–80 символов
[ ] description: 140–160 символов
[ ] reviewDate: через 6 месяцев от pubDate
[ ] tags: 4–7 штук, строчные
[ ] categories: одна из [ts-piot, kkt, markirovka, egais, zakonodatelstvo]

Стиль:
[ ] Нет слов-паразитов: является, осуществляет, в целях, таким образом
[ ] Нет эмодзи, восклицательных знаков
[ ] Активный залог, прямое обращение к читателю
```

## Стадия 5 — SEO

Запустить агент `seo-optimizer`.

**Шлюз 5:**
```
[ ] Целевой ключ в title (H1), первом абзаце, хотя бы одном H2?
[ ] description: 140–160 символов, содержит ключ + ценностный крючок?
[ ] Год «2026» в title или description?
[ ] Внутренних ссылок ≥ 3?
```

---

## Стадия 6 — КАННИБАЛИЗАЦИЯ

Сравнить `seo.keywords` новой статьи с ключами опубликованных статей.

**Шлюз 6:** пересечение < 50% с любой существующей, нет совпадения H1.

При провале — СТОП, предложить варианты A (merge) / B (differentiate) / C (canonical).

---

## Стадия 7 — ОЦЕНКА

Рассчитать итоговый балл по 10-балльной шкале:

| Критерий | Балл |
|---|---|
| Объём ≥ 1500 слов | 1 |
| Структура: 5+ H2, FAQ, заключение | 1 |
| Внутренних ссылок ≥ 3 | 1 |
| Factcheck: нет claims < 0.7 | 2 |
| SEO: ключ в title + абзаце + H2 + description OK | 2 |
| Редполитика: нет паразитов, правильный ToV | 1 |
| Hero-изображение (заполним на следующем шаге) | 1 |
| GEO: AI Citation score ≥ 60 | 1 |

**Шлюз 7:** итог ≥ 7/10. При меньшем балле — вывести список конкретных недостатков,
предложить правки, повторить проверку.

---

## Стадия 8 — HERO

Запустить генерацию hero-изображения через GitHub Actions:

```
GitHub Actions → Generate Hero Images → Run workflow
  slug: YYYY-MM-DD-slug (без .md)
```

Дождаться коммита от Actions (workflow автоматически прописывает `heroImage` во frontmatter).

**Шлюз 8:**
```
[ ] heroImage прописан в frontmatter статьи?
[ ] Файл public/images/hero/SLUG.jpg существует?
```

Если Actions недоступны — оставить heroImage пустым, добавить `# TODO: hero` в frontmatter.
Статья получает 0 за этот критерий в итоговом балле.

## Стадия 9 — ГОТОВО

Убедиться, что `draft: true` (публикация — отдельный шаг `/release-article`).

Закоммитить черновик:
```
git add src/content/blog/SLUG.md .claude/research/YYYY-MM-DD-slug-brief.md
git commit -m "draft: [title]"
git push
```

## Правила

- Никогда не пропускать стадии.
- Максимум 2 retry на стадию — затем СТОП с объяснением.
- Revision-агент получает только список конкретных проблем.
- Не устанавливать `draft: false` — это делает `/release-article`.
- Язык — только русский.
