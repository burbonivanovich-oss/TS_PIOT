# /monitor-rss — еженедельный мониторинг НПА

Читает источники из `src/content/wiki/rss-sources.md`, оценивает актуальность
для аудитории МСБ и предлагает конкретные темы.

## Аргументы

- `--cluster <name>` — `ts-piot | markirovka | kkt | egais | zakonodatelstvo`.
  Только один кластер.
- `--window 14d` (по умолчанию) — окно срочных новостей.
  - `--window 30d` — для планирования среднесрочного контента
  - `--window 60d` — для опорных статей с дедлайном через 3 месяца
  - `--window 90d` — стратегический горизонт, обзорные темы
- `--type news` (по умолчанию) — только новости с дедлайнами/изменениями
  → идут в `/plan-content` как кандидаты на новые статьи.
  - `--type guidelines` — только методички, разъяснения, письма без
    дедлайна → идут в `/maintain-content` как кандидаты на обновление
    существующих статей.
  - `--type all` — оба потока в одном отчёте с разделением.

---

## Алгоритм

### Шаг 1 — Загрузка источников

Прочитать `src/content/wiki/rss-sources.md`.
Определить список RSS-лент с их кластерами и приоритетами.

### Шаг 2 — Получение новостей

Окно по `--window` (по умолчанию 14d). Для каждой ленты:
1. Получить содержимое RSS через WebFetch.
2. Извлечь заголовки и даты за указанное окно (от сегодняшней даты).
3. Отбросить материалы старше окна и дубли по смыслу.

### Шаг 3 — Классификация и фильтрация

Каждый материал классифицировать как **news** или **guidelines**:

| Признак | news | guidelines |
|---|---|---|
| Есть конкретный дедлайн вступления в силу | да | нет |
| Меняет суммы штрафов / лимиты / ставки | да | нет |
| Меняет процедуру или порядок | да | редко |
| Подзаконный акт / приказ ведомства | часто да | возможно |
| Письмо ФНС/Минфина с разъяснением | редко да | да |
| Методичка / FAQ / комментарий регулятора | нет | да |
| Аналитический обзор / разбор кейса | нет | да |
| Изменение трактовки нормы без поправки | нет | да |

Применить фильтр по `--type`:
- `news` — только news (поток в `/plan-content`).
- `guidelines` — только guidelines (поток в `/maintain-content`).
- `all` — оба потока с разделением в отчёте.

Для каждого материала оценить актуальность:

| Критерий | Как проверить |
|---|---|
| Касается МСБ, ИП или розницы? | Ключевые слова: ИП, МСП, розница, упрощёнка, маркировка, касса |
| Есть ли конкретная дата вступления в силу? | Упоминание даты в тексте |
| Изменились ли суммы штрафов или лимиты? | Числа + слова «штраф», «лимит», «порог» |
| Затрагивает ли технические требования? | Технические термины тематики |

Отобрать **не более 10 news + 10 guidelines** на отчёт.

### Шаг 4 — Проверка дублирования

Прочитать frontmatter всех файлов в `src/content/blog/` и `src/data/content-plan.json`.
Для каждой отобранной новости измерить пересечение с существующими статьями
по совпадению нормализованных целевых ключей (lowercase, без кавычек):

- **Полное совпадение (≥ 90%)** → пометить **«обновить [slug]»**, новость
  это явное продолжение существующей темы.
- **Сильное (60–90%)** → пометить **«возможный апдейт [slug] — нужна
  ручная оценка»**. Не отбраковывать, не отправлять в «новая статья». В
  отчёте показать оба варианта (новая vs апдейт), редактор решает.
- **Слабое (< 60%)** → пометить **«новая статья»**.

Идея: автоматический порог 60% слишком жёсткий. Часто новость освещает
смежный угол существующей темы (новые сроки, штрафы, исключения) — это
по сути материал для расширения существующей статьи + новостная заметка,
а не выкид «дубль». Решение оставляем за человеком.

### Шаг 6 — Сохранение

Если пользователь подтверждает («да», «добавь», «сохрани»):

**News → контент-план:**
1. Дописать одобренные темы в `src/data/content-plan.json`
   в соответствующий раздел кластера с пометкой `# [из RSS YYYY-MM-DD]`.
2. Закоммитить: `Контент-план: темы из RSS-мониторинга DD.MM.YYYY`.

**Guidelines → maintain-queue:**
1. Дописать одобренные темы в
   `src/content/wiki/maintain-queue.md` в раздел «Кандидаты на
   обновление» с пометкой `# [из RSS-guidelines YYYY-MM-DD]`.
   Структура: `[slug] — [что добавить из методички] (источник: [URL])`.
2. Закоммитить: `Maintain-queue: кандидаты на обновление из RSS DD.MM.YYYY`.

`/maintain-content` теперь читает `maintain-queue.md` в начале своего
прохода — это первоочередные кандидаты на ревью (см. Шаг 1 в
`/maintain-content`).

---

## Использование с /loop

Для еженедельного мониторинга:
```
/loop 7d /monitor-rss
```

Агент будет запускаться раз в неделю и предлагать темы в чат.

---

## Правила

- Не добавлять темы в контент-план без явного подтверждения пользователя.
- Если RSS-лента недоступна — пропустить с пометкой `[недоступна]`, не прерывать работу.
- Язык отчёта — только русский.
- Не повторять темы, которые уже были в предыдущих отчётах (проверять по src/data/content-plan.json).
