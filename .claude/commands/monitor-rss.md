# /monitor-rss — еженедельный мониторинг НПА

Читает источники из `src/content/wiki/rss-sources.md`, оценивает актуальность
для аудитории МСБ и предлагает конкретные темы для статей.

Опционально: `--cluster <название>` — проверить только один кластер.

---

## Алгоритм

### Шаг 1 — Загрузка источников

Прочитать `src/content/wiki/rss-sources.md`.
Определить список RSS-лент с их кластерами и приоритетами.

### Шаг 2 — Получение новостей

Для каждой ленты (или только выбранного кластера):
1. Получить содержимое RSS через WebFetch.
2. Извлечь заголовки и даты за последние **14 дней** (от сегодняшней даты из CLAUDE.md).
3. Отбросить новости старше 14 дней и дубли по смыслу.

### Шаг 3 — Фильтрация

Для каждой новости оценить:

| Критерий | Как проверить |
|---|---|
| Касается МСБ, ИП или розницы? | Ключевые слова: ИП, МСП, розница, упрощёнка, маркировка, касса |
| Есть ли конкретная дата вступления в силу? | Упоминание даты в тексте |
| Изменились ли суммы штрафов или лимиты? | Числа + слова «штраф», «лимит», «порог» |
| Затрагивает ли технические требования (ТС ПИоТ, ФФД, ОФД)? | Технические термины из нашей тематики |

Отобрать **не более 10** самых релевантных новостей.

### Шаг 4 — Проверка дублирования

Прочитать frontmatter всех файлов в `src/content/blog/` и `src/content/wiki/content-plan-2026.md`.
Для каждой отобранной новости проверить: есть ли уже статья или тема в плане с пересечением > 60%?
- Если есть → пометить как «обновить [slug]» вместо «новая статья».
- Если нет → пометить как «новая статья».

### Шаг 5 — Вывод

Вывести структурированный отчёт:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 /monitor-rss — отчёт YYYY-MM-DD
Период: DD.MM–DD.MM.YYYY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## Срочно (дедлайн ≤ 30 дней)

1. [Заголовок новости]
   Источник: [название]  Дата: DD.MM.YYYY
   Кластер: [название]
   Суть: [1–2 предложения что изменилось]
   Действие: новая статья / обновить [slug]
   Предлагаемый целевой запрос: "..."

## Актуально (дедлайн 30–90 дней или без дедлайна)

2. ...

## Не актуально / дубль

- [заголовок] → дубль [slug]
- [заголовок] → не касается МСБ

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Итого предложений: N новых статей, M обновлений
Следующий шаг: запустить /publish-article "<запрос>" для срочных тем
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Шаг 6 — Сохранение в контент-план (опционально)

Если пользователь подтверждает («да», «добавь», «сохрани»):
1. Дописать одобренные темы в `src/content/wiki/content-plan-2026.md`
   в соответствующий раздел кластера с пометкой `# [из RSS YYYY-MM-DD]`.
2. Закоммитить: `git add src/content/wiki/content-plan-2026.md && git commit -m "Контент-план: темы из RSS-мониторинга DD.MM.YYYY"`.

---

## Использование с /loop

Для еженедельного мониторинга:
```
/loop 7d /monitor-rss
```

Агент будет запускаться раз в неделю и предлагать темы в чат.

---

## Правила

- Не добавлять темы в контент-план без явного подтверждения пользователя.
- Если RSS-лента недоступна — пропустить с пометкой `[недоступна]`, не прерывать работу.
- Язык отчёта — только русский.
- Не повторять темы, которые уже были в предыдущих отчётах (проверять по content-plan-2026.md).
